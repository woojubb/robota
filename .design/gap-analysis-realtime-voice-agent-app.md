# Gap Analysis — 실시간 다중 페르소나 음성 앱에서 Robota 대신 Vercel AI SDK를 택한 이유

> **작성일**: 2026-07-01
> **작성 맥락**: 별도 프로젝트(`dev/speech` — 실시간 음성 토론/발표 트레이닝 앱)의 LLM/에이전트 계층 기술 선정 과정에서 도출.
> **문서 목적**: "Vercel이 낫다"는 결론 자체가 아니라, **이 유스케이스에서 Robota의 어떤 공백이 탈락 요인이 되었는지**를 기록해 Robota 확장 로드맵의 입력으로 삼기 위함.
> **주의**: 아래 "근거"의 코드 경로는 README/공개 API/소스 grep 기반의 스냅샷(`3.0.0-beta`)이다. 세부 구현은 소스로 재확인 필요(⚠️ 표기).
>
> **전제 정정(중요)**: Robota는 `agent-cli`(코딩 어시스턴트) 이전에, **조립형(composable) 에이전트 라이브러리**다 — `agent-core`/`agent-provider`/`agent-session`/`agent-transport-*` 등의 블록을 골라 조립해 임의의 에이전트 앱을 만드는 것이 설계 의도다. 따라서 이 문서의 "공백"은 _"Robota가 코딩 전용이라 못 한다"_ 가 아니라, **"조립형 라이브러리인데 이 앱을 조립하려니 아직 없는 블록이 있다"** 로 읽어야 한다. 각 공백은 곧 **추가되면 좋은 조립 블록**의 명세다.

---

## 0. 한 줄 결론

우리가 필요한 것은 **"시스템 프롬프트로 LLM을 스트리밍 호출하고, 코치 리포트를 스키마로 강제 받는" 얇고 상태 없는 agent 호출 + 커스텀 오디오 실시간 채널**이다. Robota는 조립형 라이브러리이므로 이론상 조립 가능하지만, 이 앱을 조립하는 데 필요한 **(1) first-class 구조화 출력, (2) 바이너리/오디오를 나르는 범용 전송, (3) 하나의 대화록을 공유하는 다중 에이전트 "룸" 모델**의 조립 블록이 (아직) 없다. 반면 이미 조립돼 있는 블록(세션/권한/훅, subagent 격리, preset, 툴 인프라)은 이 앱에선 대부분 쓰지 않는다. 결국 "없는 블록은 직접 만들고, 있는 블록은 안 쓰는" 상태가 되어, LLM 계층은 Vercel AI SDK v6 + AI Gateway로 택했다.

**이것은 Robota의 품질 평가가 아니라 유스케이스 적합성 분석이다.** 조립 대상이 코딩/작업 자동화 에이전트였다면 이미 있는 블록이 그대로 맞아 결론은 반대였을 것이다(§5 참조).

## 1. 평가한 유스케이스

- 브라우저 웹 앱. **실시간 음성** 대화(턴 기반, barge-in 불필요).
- 한 "방"에 **AI 페르소나 2~3명**(성격·역할·목소리 상이) + 사용자 1명.
- 파이프라인: 사용자 마이크 → **STT(한국어, Deepgram)** → **Director LLM**(누가/어떤 순서로 답할지) → **Persona LLM**(캐릭터별 응답) → **TTS(ElevenLabs, 페르소나별 다른 목소리)** → 재생.
- 별도 **Coach LLM**이 대화록을 관찰 → 실시간 힌트 + 종료 시 **구조화 평가 리포트**.
- 페르소나는 `.md`/`.json` 파일로 정의(코드 수정 없이 추가).

즉 agent 관점에서 필요한 것은 3종의 역할 에이전트(Director/Persona/Coach)와, 이들이 **하나의 공유 대화록**을 두고 협업하는 구조, 그리고 그 결과를 **오디오로** 실어 나르는 실시간 채널이다.

## 3. Vercel 채택으로 이어진 5가지 공백

### G1. First-class 구조화 출력의 부재

- **관찰**: 코치 리포트는 "논리/근거/반박/설득력/전달력" 등 **고정 스키마 JSON**으로 매번 안정적으로 받아야 한다. AI SDK는 `generateText({ output: Output.object({ schema }) })`로 스키마 위반 시 모델이 자동 재시도한다.
- **근거(⚠️ 재확인 필요)**: `packages/agent-core`·`agent-provider`에 `response_format`/`json_schema`/`Output.object` 상당물이 보이지 않음. `outputSchema`는 `tool-registry/function-tool`의 **툴 파라미터** 맥락. core/provider에 `zod` 미사용. → 구조화 출력을 하려면 "툴 호출"로 우회해야 하고, 자유 발화 응답과 스키마 응답을 한 API에서 매끄럽게 오가기 어렵다.
- **영향**: 코치 리포트/실시간 힌트를 타입 안전하게 뽑는 비용이 커짐. 이 앱에서 반복적으로 필요한 기능이라 가중치가 높았다.
- **Robota 확장 제안**: 에이전트 레벨에서 `run(prompt, { output: schema })` 형태의 **스키마 강제 + 자동 재시도** 구조화 출력 API. Zod/JSON-Schema 어댑터. 각 프로바이더의 native structured output(예: Anthropic `structuredOutputMode`, OpenAI response_format)로 매핑.

### G2. 전송(transport)이 텍스트-에이전트 프로토콜에 결합

- **관찰**: 우리 실시간 채널은 **마이크 오디오 청크(업)·TTS 음성 클립(다운)·자막·코칭·발화자 상태**를 나른다. `agent-transport-ws`는 `text_delta`/`abort` 등 텍스트 에이전트 대화용 와이어 프로토콜이다.
- **근거(⚠️)**: `agent-transport-ws/README` + `examples/websocket-chat`가 "클라이언트당 세션 + 텍스트 델타 스트리밍" 패턴. `agent-interface-transport/src/interaction-contracts.ts`에 audio/binary 키워드가 grep에 걸리나 실제로 바이너리 오디오 스트림 반송을 지원하는지는 미확인. 사실상 우리가 커스텀 프로토콜을 새로 짜야 하므로 Robota WS의 재사용 이점이 소멸.
- **영향**: "WS 예제 재사용"이 초기엔 매력이었으나, 오디오/커스텀 이벤트를 얹으면 프레임워크 프로토콜과 충돌 → 순 손해.
- **Robota 확장 제안**: 전송 계층을 **페이로드 불가지론적**으로. (a) 바이너리/오디오 프레임 채널, (b) 사용자 정의 이벤트 타입 등록, (c) 텍스트-델타 프로토콜과의 분리. 이게 있으면 음성/멀티모달 앱이 Robota 위에서 가능해진다.

### G3. 다중 에이전트가 공유하는 "룸/공유 대화록" 프리미티브 부재

- **관찰**: 우리 모델은 **1 방 = 3 에이전트 + 1 공유 append-only 대화록**이며, Director가 그 대화록을 읽어 발화 순서를 정하고 각 Persona가 같은 대화록에 기여한다.
- **근거(⚠️)**: `agent-session`/`InteractiveSession`은 "클라이언트당 하나의 에이전트 대화" 지향(`websocket-chat` = "one InteractiveSession per connected client"). N개 에이전트가 하나의 대화록을 공유하는 1급 개념은 관찰되지 않음.
- **영향**: 어차피 우리가 `ConversationStore`(단일 진실 소스)와 오케스트레이터를 직접 만든다 → Robota 세션 계층이 이점이 아니라 우회 대상이 됨. AI SDK는 세션을 강제하지 않아(경량 호출) 오히려 자유롭다.
- **Robota 확장 제안**: **Room/Ensemble** 프리미티브 — 여러 에이전트가 공유 대화록을 구독/기여하고, **턴 선택(Director) 훅**을 꽂을 수 있는 구조. 이것은 코딩 subagent(격리·병렬)와는 반대 방향(공유·순차)의 새 유스케이스를 연다.

### G4. "그냥 LLM 호출" 경량 경로의 문맥 지향성

- **관찰**: 우리는 툴·권한·프로젝트 컨텍스트 없이 순수 프롬프트 호출만 원한다.
- **공정한 인정**: Robota에 `createQuery`(`agent-framework/src/query.ts`)라는 경량 진입점이 **있다.** 다만 프레임워크가 config/context 로딩 등 코딩-에이전트 조립을 전제로 설계되어, "무상태 순수 호출"로 쓰기엔 주변 조립 비용이 있다.
- **영향**: 결정적 요인은 아니나 G1~G3와 합쳐져 "얇게 쓰기엔 무겁다"는 인상을 강화.
- **Robota 확장 제안**: 세션/권한/툴 스캐폴딩 없이 `provider + systemPrompt + stream`만 필요한 **`streamText` 상당의 무상태 헬퍼**를 1급으로 노출.

### G5. 오디오 멀티모달(STT/TTS) 어댑터 계약 부재

- **관찰**: 이 앱의 본질은 음성. 어느 프레임워크도 STT/TTS를 직접 안 하지만, **어댑터 계약(interface)** 이라도 있으면 생태계가 붙는다.
- **공정한 인정**: Robota는 Gemini 경유 **이미지** 멀티모달(`agent-provider/src/gemini/image-operations.ts`)은 있음. 그러나 **오디오 입력/출력, STT/TTS 개념은 없음.**
- **Robota 확장 제안**: `ISttAdapter`/`ITtsAdapter` 계약 + 스트리밍 오디오 타입 정의. 구현체(Deepgram/ElevenLabs 등)는 커뮤니티가 채우게. G2(오디오 전송)와 결합하면 Robota가 **음성 에이전트 프레임워크**로 확장된다.

## 4. 부차적 요인 (결정 가중치 낮음)

- **안정성**: AI SDK v6 stable vs Robota `3.0.0-beta`(API 변동 고지). 프로덕션 지향 앱에서 소폭 감점.
- **UI 렌더**: 자막/리포트 마크다운 스트리밍에 AI Elements가 즉시 활용 가능. Robota `agent-web`은 코딩 지향.
- **관측성**: AI Gateway가 비용추적·failover·로그를 기본 제공. Robota는 플러그인 + 자체 구성 필요(다른 접근, 절대적 단점은 아님).

## 5. Robota에 이미 조립돼 있는 강한 블록 (균형)

조립 대상이 **코딩/작업 자동화 에이전트**였다면 아래 블록들이 그대로 맞아 Robota가 유리했을 지점(= 조립형 라이브러리로서 이미 갖춘 자산):

- **preset 시스템**(`agent-preset`) — 이름 붙은 페르소나·모델·행동 번들. 페르소나 카드 개념과 실제로 잘 맞았다(이번엔 파일 로더로 대체했을 뿐).
- **세션/권한/훅**(`agent-session`) — 툴 실행 승인, 라이프사이클 훅, compaction.
- **subagent 격리**(`agent-subagent-runner`) — 자식 프로세스 + worktree 격리. 병렬 코딩 작업엔 강력.
- **툴 인프라 + 내장 툴 8종**, 코딩 어시스턴트(`agent-cli`) 완성도.
- **DAG 오케스트레이션 계열**(`dag-*`) — 복잡한 워크플로 실행. (이 앱의 턴 조율엔 오버킬이라 미채택했지만, 확장 시 Director를 DAG로 표현하는 선택지는 존재.)

## 6. 결정을 뒤집을 조건 (Robota 재검토 트리거)

다음이 갖춰지면 이 부류의 앱에서 Robota가 강력한 후보가 된다:

1. **G1** 에이전트 레벨 스키마 강제 구조화 출력.
2. **G2** 페이로드 불가지론 전송(바이너리/오디오 + 커스텀 이벤트).
3. **G3** 공유 대화록 기반 다중 에이전트 Room + 턴 선택 훅.

이 셋이 핵심이다. G4/G5는 있으면 결정적 가산점.

## 8. 부록 — 조사 방법 & 한계

- 방법: `README.md`, 각 패키지 `README`, 공개 API 표면, 소스 grep(구조화 출력/zod/audio/binary/경량 호출/멀티모달) — 스냅샷 `3.0.0-beta`(monorepo `0.1.0`).
- 한계: 내부 구현 세부(특히 `interaction-contracts.ts`의 audio 관련 실제 지원 범위, `createQuery`의 실제 무게)는 소스로 재확인 필요(⚠️).
- 이 문서는 **하나의 유스케이스(실시간 다중 페르소나 음성 앱)** 관점의 적합성 분석이며, 프레임워크 전반의 우열 판정이 아니다.
