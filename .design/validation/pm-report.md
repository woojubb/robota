## 1. 제품 평가 개요

### 포지셔닝 요약

`@robota-sdk/agent-cli`는 "AI 코딩 어시스턴트 CLI" 카테고리에서 **BYOK + 멀티 프로바이더 + 오픈소스(MIT)**라는 세 가지 축을 핵심 차별점으로 내세운다. Claude Code가 Anthropic 독점, Cursor가 IDE 종속인 상황에서 "어떤 모델이든, 어떤 환경이든, 내가 소유한 도구"라는 포지션은 이론적으로 명확하다.

### 현재 상태의 솔직한 평가

**강점:**

- 아키텍처적으로 견고하다. CLI가 TUI 레이어에 불과하고 `agent-framework`가 비즈니스 로직을 담는 계층 분리는 올바른 설계다.
- 멀티 프로바이더 지원이 실제로 구현되어 있다. 설정 파일 한 줄 변경으로 전환 가능하다는 주장이 코드 레벨에서 뒷받침된다.
- 엔터프라이즈 기반(org-policy, RBAC, audit log)이 이미 MVP 수준으로 존재한다. 경쟁사 대비 이른 시점에 이 레이어를 갖췄다.
- SDK 임베딩 가능성이 진짜 차별점이다. `@robota-sdk/agent-framework`를 import해서 자기 앱에 에이전트 런타임을 넣을 수 있다는 것은 다른 경쟁사 어디도 제공하지 않는다.

**약점 (솔직하게):**

- Node.js 22+ 요구는 npx 첫 실행에서 상당수 개발자를 막는다. npm 생태계의 주류는 여전히 Node 20 LTS다. 첫 번째 마찰이 "당신의 Node.js 버전이 너무 낮습니다"라면 이탈률이 높다.
- macOS Terminal.app에서 CJK IME 크래시는 해결되지 않은 알려진 버그다. 한국어를 쓰는 팀이 만든 도구에서 한국어 입력이 기본 터미널에서 죽는다는 것은 아이러니이자 신뢰 손상 요소다.
- 커뮤니티 트랙션이 0에 가깝다. Showcase 페이지에 커뮤니티 프로젝트가 없다. GitHub Discussions도 활성화 전이다.
- 공개 beta 상태지만 사용자가 무엇을 기대해야 하는지 불명확하다. "v1.0.0은 언제?" 에 대한 답이 "Q3 2026"이라는 로드맵이 있지만, 그 기준이 "P0 버그 해결 + 핵심 여정 검증"으로 추상적이다.
- 세션 자동 이름 부여, /cost 개선 같이 UX에 직접 영향 주는 기능이 아직 계획 중이다. 기본적인 편의 기능이 beta.68을 기다리고 있다는 것은 사용자 경험 완성도가 낮다는 신호다.

### 타겟 사용자

구조적으로 가장 잘 맞는 사용자는 **"AI 코딩 도구의 비용이나 잠금 효과(lock-in)에 불만을 가진 TypeScript 개발자"**다. 그러나 실제 채택 경로는 CLI 사용자와 SDK 임베더로 나뉘며, 이 두 그룹의 필요가 다르다.

### 핵심 가치 제안 분석

| 가치 제안                      | 실현 가능성 | 현재 증거              |
| ------------------------------ | ----------- | ---------------------- |
| "어떤 프로바이더든 one config" | 높음        | 코드 검증됨            |
| "월구독 없이 토큰 비용만"      | 높음        | 구조적으로 맞음        |
| "MIT, 포크해서 수정 가능"      | 높음        | 라이선스 명확          |
| "SDK로 내 앱에 임베딩"         | 중간        | API 존재, 예제 부족    |
| "30초 설치"                    | 낮음        | Node 22 장벽, CJK 버그 |
| "CI/CD에서 스크립트처럼 사용"  | 중간        | 기능 있음, 문서 부족   |

## 2. 검증 전략

### 핵심 질문

"실제 개발자가 이 도구를 설치하고, 첫 번째 유용한 결과를 얻고, 다음 날도 다시 쓸 것인가?"

### 검증 메트릭

**활성화 메트릭 (Activation Metrics):**

- Time to First Value(TTFV): `npx @robota-sdk/agent-cli` 실행 후 첫 번째 AI 응답까지의 시간
- 목표: 2분 이내 (현재 추정: Node 버전 이슈 없을 경우 3-4분)
- 설치 완료율: npx 실행 후 provider 설정까지 완료한 비율
- 첫 메시지 전송률: 설정 후 실제로 프롬프트를 입력한 비율

**리텐션 메트릭 (Retention Metrics):**

- D1 리텐션: 설치 다음 날 재사용 비율 (목표: 40%+)
- Weekly Active Users / Total Installs 비율
- 세션 평균 지속 시간 (현재 측정 불가 — 텔레메트리 없음)

**성장 메트릭 (Growth Metrics):**

- npm 주간 다운로드 추이
- GitHub Stars 증가율
- 이슈/PR 제출 비율 (커뮤니티 참여도)
- Showcase 커뮤니티 프로젝트 수

**중요 한계:** 현재 텔레메트리가 없어서 활성화/리텐션 메트릭을 직접 측정할 수 없다. "No telemetry"가 프라이버시 강점이지만 제품 개선을 어렵게 만드는 트레이드오프다. opt-in 익명 사용 통계라도 제공할지 결정이 필요하다.

### 검증 단계

#### Alpha 단계 (현재 ~ beta.70)

**목표:** 핵심 사용 경로가 crash 없이 동작하는가

검증 방법:

- 내부 팀 구성원 5-10명이 실제 프로젝트에서 매일 사용
- 매주 "사용하다 막힌 점" 비구조화 인터뷰
- 이슈 트래커에서 크래시/오류 패턴 분류

성공 기준:

- Node 20 환경 진입 장벽 제거 또는 명확한 안내
- macOS Terminal.app CJK 크래시 해결
- `npx @robota-sdk/agent-cli` → 첫 응답까지 2분 이내

#### Beta 단계 (beta.70 ~ v1.0.0-rc)

**목표:** 외부 개발자 10명이 독립적으로 설치하고 쓸 수 있는가

검증 방법:

- TypeScript 개발자 10-20명 대상 비공개 베타 초청 (GitHub Discussions를 통해)
- 설치 과정 화면 녹화 요청 (Loom 등)
- 1주일 후 "아직 쓰고 있나?" 팔로업

성공 기준:

- 10명 중 7명이 첫 설치 성공 (단일 블로커 없음)
- 10명 중 5명이 1주일 후에도 사용 중
- 치명적 버그(데이터 손실, 크래시) 0건

#### GA 단계 (v1.0.0)

**목표:** 오픈소스 커뮤니티에서 자생적 성장이 시작되는가

검증 방법:

- Product Hunt 런칭
- Hacker News 제출 (Show HN)
- 첫 1개월 npm 다운로드 추이 모니터링

성공 기준:

- npm 주간 다운로드 500+
- GitHub Stars 200+
- 커뮤니티에서 제출한 Showcase 프로젝트 3개 이상

## 3. 사용자 페르소나

### 페르소나 1: 비용 절감 추구자 (Cost-Conscious Developer)

**누구:** Claude Code나 Cursor를 쓰다가 월 $20-40 비용에 부담을 느끼는 프리랜서 또는 사이드 프로젝트 개발자

**왜 Robota를 쓸 것인가:** 구독 없이 직접 API 키를 쓰면 토큰 비용만 내면 된다. DeepSeek 같은 저렴한 모델로 전환하면 더 싸다.

**어떤 상황에서:** 개인 프로젝트 작업 시, 또는 회사에서 Claude Code 라이선스를 제공하지 않을 때

**검증 포인트:** 설치 및 provider 전환이 쉬운가? 비용 트래킹(/cost 명령)이 잘 작동하는가?

**현재 갭:** /cost 명령 개선이 beta.68 예정이다. 세션당 비용 추적이 불완전한 상태다.

### 페르소나 2: 멀티 모델 실험자 (Model Agnostic Experimenter)

**누구:** 프로젝트에 따라 Claude, GPT-4, DeepSeek, 로컬 Ollama를 번갈아 쓰고 싶은 개발자

**왜 Robota를 쓸 것인가:** 설정 파일 한 줄로 모델 전환. 도구를 바꿀 필요 없다.

**어떤 상황에서:** 새 모델이 나왔을 때 테스트, 비용-품질 trade-off 실험, 특정 모델이 특정 작업에 더 나은지 비교

**검증 포인트:** provider hot-swap이 재시작 없이 작동하는가? (현재 done 상태이지만 컨텍스트 유지가 관건)

**현재 갭:** provider 전환 시 세션 재시작이 필요했던 문제가 CLI-025로 해결됐다. 하지만 전환 UX의 완성도 검증이 필요하다.

### 페르소나 3: 보안/프라이버시 중시 개발자 (Privacy-First Developer)

**누구:** 코드를 외부 서버에 올리고 싶지 않은 금융, 법무, 의료 도메인 개발자 또는 기업 보안 정책이 엄격한 환경의 개발자

**왜 Robota를 쓸 것인가:** 완전 로컬 실행(Ollama/LM Studio), 텔레메트리 없음, MIT 소스 감사 가능, 에어갭 배포 지원

**어떤 상황에서:** 사내 코드베이스 분석, 규제 환경의 자동화, 기업 보안팀이 외부 AI 서비스를 막아둔 환경

**검증 포인트:** Ollama 연결이 설정 없이 즉시 되는가? 오프라인 환경에서 동작하는가?

**현재 갭:** "로컬 모델 first-class"라고 주장하지만 실제 Ollama 설정 과정의 UX 검증이 필요하다. LM Studio 연결 가이드가 Getting Started에 있는 것은 좋다.

### 페르소나 4: SDK 임베더 (SDK Embedder)

**누구:** 자기 SaaS/앱/CLI에 AI 코딩 어시스턴트 기능을 넣으려는 개발자. 처음부터 만들기 싫어서 엔진만 빌려 쓰고 싶다.

**왜 Robota를 쓸 것인가:** `@robota-sdk/agent-framework`를 import하면 이미 검증된 세션 관리, 권한 시스템, 멀티 프로바이더를 얻는다.

**어떤 상황에서:** 인디 개발자가 AI 코딩 도구를 만들 때, 기업이 내부용 AI 어시스턴트를 구축할 때

**검증 포인트:** SDK API가 안정적인가? 문서화가 충분한가? 예제가 실제로 돌아가는가?

**현재 갭:** Getting Started에 SDK 예제가 있지만 "다음 단계" 문서가 부실하다. Next.js, Express 임베딩 예제가 Q3로 밀려 있다.

### 페르소나 5: CI/CD 자동화 엔지니어 (Automation Engineer)

**누구:** PR 리뷰, 코드 품질 검사, 문서 생성 등을 AI로 자동화하려는 DevOps/플랫폼 엔지니어

**왜 Robota를 쓸 것인가:** `robota -p "..." --output-format json`으로 헤드리스 실행, GitHub Actions에서 스크립트처럼 사용 가능

**어떤 상황에서:** PR 자동 리뷰 봇, 코드 변경 요약 생성, 문서 자동 업데이트 파이프라인

**검증 포인트:** 헤드리스 모드 출력이 예측 가능한가? 비인터랙티브 환경에서 프롬프트 없이 실행되는가?

**현재 갭:** CI/CD 통합 문서가 CLI-021로 recently done 처리됐다. 실제 GitHub Actions workflow 예제의 정확성 검증 필요.

## 4. 핵심 사용 시나리오 (User Stories)

### 시나리오 A: 첫 번째 사용 (모든 페르소나 공통)

```
As a 개발자,
When 나는 npx @robota-sdk/agent-cli를 실행하고,
Then 2분 안에 AI와 첫 대화를 할 수 있어야 한다.
```

**현재 지원 여부:** 부분 지원  
**갭:** Node 22 요구로 Node 20 사용자 차단, CJK IME 크래시(macOS), API 키 없는 사용자의 LM Studio 연결 UX 불완전

### 시나리오 B: 프로젝트 컨텍스트 활용

```
As a TypeScript 개발자,
When 내 프로젝트 폴더에서 robota를 실행하면,
Then AGENTS.md/CLAUDE.md를 자동으로 읽어 프로젝트 규칙을 이해하고,
And "이 프로젝트의 아키텍처를 설명해줘"에 정확히 답해야 한다.
```

**현재 지원 여부:** 지원  
**갭:** 없음 (Context Discovery 기능 완성). 단, 컨텍스트 크기가 70%/90%를 넘을 때의 UX (경고 배너, /compact) 완성도 확인 필요

### 시나리오 C: 모델 전환

```
As a 멀티 모델 실험자,
When 나는 세션 중간에 /provider switch openai를 실행하면,
Then 이전 대화 컨텍스트를 잃지 않고 OpenAI로 전환되어야 한다.
```

**현재 지원 여부:** done (CLI-025)  
**갭:** 구현됐다고 표시되어 있지만 실제 e2e 검증 증거가 없음. provider hot-swap은 아키텍처적으로 복잡한 기능이라 실제 동작 검증 필요

### 시나리오 D: GitHub Actions에서 자동화

```
As a DevOps 엔지니어,
When git diff HEAD~1 | robota -p "Review this diff"를 GitHub Actions에서 실행하면,
Then 프롬프트 없이 JSON 형식으로 리뷰 결과가 출력되어야 한다.
```

**현재 지원 여부:** 지원  
**갭:** 기능은 있지만 공식 GitHub Action이 없다 (Q3 예정). 직접 구성하는 예제 문서가 최근 추가됐지만 커뮤니티 검증이 없음

### 시나리오 E: SDK 임베딩

```
As a SaaS 개발자,
When 나는 @robota-sdk/agent-framework를 내 Express 앱에 설치하면,
Then 10줄 이하의 코드로 AI 코딩 어시스턴트 세션을 내 앱에 넣을 수 있어야 한다.
```

**현재 지원 여부:** 부분 지원  
**갭:** API는 존재. Getting Started에 SDK 예제 있음. Express/Next.js 임베딩 레퍼런스 구현이 Q3 예정. 현재 사용자가 혼자 이 경로를 찾아가기 어렵다.

### 시나리오 F: 로컬 모델 (인터넷 없는 환경)

```
As a 보안 중시 개발자,
When 인터넷 없는 환경에서 LM Studio를 실행하고 robota를 시작하면,
Then API 키 없이 로컬 모델로 즉시 사용 가능해야 한다.
```

**현재 지원 여부:** 이론적 지원 (설정 파일로 구성 가능)  
**갭:** Getting Started에 "No API key?" 섹션이 있지만 LM Studio 연결 과정의 실제 UX 검증이 부족. 자동 감지(localhost 로컬 서버)가 없어 수동 설정 필요

## 5. 온보딩 경험 분석

### 설치 → 첫 가치까지의 흐름 (현재)

```
[1] npx @robota-sdk/agent-cli
    → (장벽 1) Node < 22이면 즉시 실패. 오류 메시지 수준이 관건.
    → 설치 시간: ~30초 (의존성 크기에 따라 다름)

[2] 첫 실행: Provider 선택 인터랙티브 프롬프트
    → 지원 프로바이더 목록 표시 (Anthropic, OpenAI, DeepSeek, Gemma, Qwen)
    → API 키 입력 (마스킹 처리)
    → 언어 선택 (ko/en/ja/zh)
    → ~/.robota/settings.json 생성

[3] TUI 실행
    → (장벽 2) macOS Terminal.app + CJK IME → 크래시
    → 정상 환경: 입력창, StatusBar, 슬래시 자동완성 사용 가능

[4] 첫 메시지
    → 프로젝트 폴더에서 실행 시 AGENTS.md 자동 로딩
    → 실제 AI 응답 스트리밍
    → Aha Moment: "내 프로젝트를 이해하는 AI 어시스턴트가 터미널에 있다"
```

### 마찰 지점 분석

**블로커 (이탈 확실):**

1. **Node 22 요구** — npm 생태계의 상당수가 Node 20 LTS. `npx` 실행 즉시 오류. 친절한 안내 메시지라도 있어야 한다.
2. **macOS Terminal.app CJK 크래시** — 한국/일본/중국 개발자 전체 이탈. iTerm2 사용 권장은 해결책이 아니다.

**마찰 (이탈 가능성 높음):** 3. **API 키 없는 경우의 UX** — LM Studio 사용 경로가 있지만 발견성이 낮다. 첫 화면에 "API 키 없으세요? 로컬 모델로 시작하기" 경로가 명확해야 한다. 4. **update 확인 인터랙션** — 시작 시 npm 업데이트 확인이 느릴 수 있다. `--disable-update-check` 플래그가 있지만 첫 사용자는 모른다.

**불편함 (경험 저하):** 5. **세션 이름이 자동으로 없음** — 세션 목록이 ID만 표시. auto-naming이 beta.68 예정. 복수 세션 관리가 불편하다. 6. **비용 추적 불완전** — /cost 명령이 있지만 세션당 비용 집계가 beta.68 개선 예정. 비용 절감이 핵심 가치인데 비용 추적이 약하다. 7. **온보딩 문서 없는 빈 상태** — Getting Started에 코드 예제가 많지만 "첫 번째로 뭘 물어봐야 하나?" 가이드가 없다. Claude Code의 "Try asking..." 프롬프트 제안 같은 것이 없다.

### Aha Moment 분석

현재 Aha Moment는 **"AGENTS.md를 인식하고 프로젝트를 이해하는 첫 응답"**이다. 이것은 진짜 가치이지만, 이 경험에 도달하려면 AGENTS.md가 있는 프로젝트에서 실행해야 한다. 신규 사용자는 AGENTS.md가 없는 빈 폴더에서 시작하는 경우가 많다. "이 프로젝트에 AGENTS.md가 없습니다. `robota init`으로 시작해보세요"같은 안내가 Aha Moment까지의 경로를 단축시킬 수 있다.

좋은 소식: `robota init` 명령이 recently done 처리됐다 (beta.67 roadmap 기준). 이것의 UX가 잘 되어있다면 온보딩 개선에 크게 기여한다.

## 6. 경쟁 분석

### 경쟁 구도

| 도구        | 모델        | 배포            | 가격          | 임베딩   | 오픈소스 |
| ----------- | ----------- | --------------- | ------------- | -------- | -------- |
| Claude Code | Anthropic만 | 클라우드        | $20/월 + 토큰 | 불가     | 일부     |
| Cursor      | 멀티        | IDE 플러그인    | $20/월        | 불가     | 불가     |
| Aider       | 멀티        | CLI             | BYOK          | 불가     | MIT      |
| Cline       | 멀티        | VSCode 사이드바 | BYOK          | 불가     | MIT      |
| **Robota**  | **멀티**    | **CLI + SDK**   | **BYOK**      | **가능** | **MIT**  |

### Claude Code 대비

**Robota의 강점:**

- 멀티 프로바이더 (Claude Code = Anthropic 독점)
- BYOK, 구독 없음 (Claude Code = $20/월 구독)
- SDK 임베딩 가능 (Claude Code는 불가)
- 로컬 모델 지원 (Claude Code는 불가)
- 설정 파일 호환: `.claude/settings.json`을 그대로 읽는다

**Claude Code의 강점:**

- Claude 모델 최적화 (프롬프트, 도구 호출 정확도)
- Anthropic 공식 지원
- 더 큰 커뮤니티, 더 많은 문서, 더 많은 스킬/플러그인
- 완성도 높은 UX (다년간 다듬어진 제품)
- 소비자 신뢰 (Anthropic 브랜드)

**Robota가 Claude Code 사용자를 가져올 수 있는 조건:**
"Anthropic만 써야 한다는 제약이 싫거나, 로컬 모델이 필요하거나, 내 도구에 같은 엔진을 넣고 싶은" 사용자. 이 집합은 실재하지만 규모 추정이 어렵다.

### Aider 대비

Aider는 Python 기반 멀티 프로바이더 CLI로 가장 직접적인 경쟁자다.

**Robota의 강점:**

- TypeScript 에코시스템 네이티브
- SDK 임베딩 가능
- 더 풍부한 TUI (Aider는 기본 터미널 UI)
- 플러그인 아키텍처

**Aider의 강점:**

- 더 성숙한 제품 (수년간 운영, 수천 GitHub Stars)
- git 통합이 강력 (커밋 자동화, diff 기반 워크플로우)
- 더 큰 커뮤니티

**결론:** Robota가 Aider보다 잘할 수 있는 것은 TypeScript 개발자 타겟팅과 SDK 임베딩이다. Aider의 git 워크플로우 강점은 Robota에서 아직 약하다.

### Cline 대비

Cline은 VSCode 익스텐션이므로 직접 경쟁보다는 "터미널 vs IDE" 선택의 문제다. CLI를 선호하는 개발자는 Cline 대신 Robota를 쓸 이유가 있다.

### 어떤 사용자가 Robota를 선택할 것인가

현실적으로 Robota를 선택할 가능성이 가장 높은 사용자:

1. TypeScript/Node.js 개발자로서 AI 도구를 코드에 임베딩하고 싶은 경우
2. 기업 보안 정책으로 외부 AI 서비스를 못 쓰는 환경에서 로컬 모델이 필요한 경우
3. 여러 AI 모델을 비교/전환하며 사용하는 파워 유저
4. Claude Code를 쓰다가 Anthropic 락인이 싫어진 개발자

Robota가 주류 개발자에게 선택받으려면 "Claude Code보다 좋다"가 아니라 "Claude Code가 해결 못 하는 문제를 해결한다"는 포지셔닝이 필요하다.

## 7. 성장 전략과 방향성

### 핵심 전제

Robota는 두 가지 방향으로 성장할 수 있다:

- **A경로:** 개인 개발자 대상 Claude Code 대안 CLI
- **B경로:** 개발자가 AI 도구를 만드는 데 쓰는 SDK 플랫폼

이 두 방향은 서로 배타적이지 않지만 초기 리소스 집중 방향이 달라야 한다. **현재 마케팅은 A경로를 강조하고, 아키텍처는 B경로를 더 잘 지원한다.** 이 불일치를 해결해야 한다.

내 의견: **B경로(SDK 플랫폼)가 더 큰 기회다.** CLI는 Claude Code와 정면 경쟁해야 하고 그 게임은 이기기 어렵다. SDK 임베딩은 다른 경쟁자가 없는 블루오션이다.

## 9. PM 관점 백로그 제안

아래는 기술적 기능 외에 제품 관점에서 필요한 개선 사항이다. 현재 CLI-\* 백로그가 기술 버그 중심인 데 반해, 이것은 사용자 채택과 성장 관점이다.

### PM-001: 온보딩 체크리스트 및 첫 사용 가이드

**사용자 가치:** 처음 사용자가 첫 5분 안에 의미 있는 결과를 얻는다  
**우선순위:** P0  
**내용:** 첫 실행 시 "이런 것을 물어보세요" 프롬프트 제안. `robota init`이 AGENTS.md를 만들 때 "이제 robota에서 '이 프로젝트의 주요 파일들을 알려줘'를 입력해보세요" 안내 출력.

### PM-002: opt-in 익명 사용 통계 (텔레메트리)

**사용자 가치:** 팀이 실제 사용 패턴을 알고 더 좋은 제품을 만든다  
**우선순위:** P1  
**내용:** 설치 시 "개선을 위한 익명 사용 데이터 수집에 동의하시겠습니까?" 프롬프트. 수집 항목: 실행 횟수, 사용 명령, 프로바이더 종류(모델명/키 제외), 오류 코드. 언제든 비활성화 가능.

### PM-003: 프로바이더별 비용 계산기 통합

**사용자 가치:** "이 작업에 DeepSeek가 Claude보다 10배 저렴하다"는 것을 세션 중에 확인한다  
**우선순위:** P1  
**내용:** /cost 명령에 모델별 토큰 단가 기반 비용 계산. 세션 완료 시 "이 대화: Anthropic Claude Sonnet = $0.12, DeepSeek V3 = $0.008 (94% 절감)"처럼 비교 표시.

### PM-004: 신규 사용자 대상 "Try Robota" 대화형 데모 (웹)

**사용자 가치:** API 키 없이도 Robota의 실제 경험을 해볼 수 있다  
**우선순위:** P1  
**내용:** robota.io에 웹 기반 샌드박스 데모. 제한된 샘플 프로젝트(공개 레포)에서 robota CLI를 시뮬레이션. 실제 모델 호출 없이 사전 녹화된 시나리오로도 가능.

### PM-005: 공식 GitHub Action (robota-sdk/action@v1)

**사용자 가치:** PR 리뷰 자동화를 10줄 YAML로 시작한다  
**우선순위:** P1  
**내용:** GitHub Marketplace에 등록된 공식 Action. `git diff | robota -p "Review this"` 패턴을 캡슐화. secret으로 API 키 주입.

### PM-006: Showcase 커뮤니티 프로젝트 적극 유치

**사용자 가치:** "실제로 이걸로 뭘 만들었나"를 보고 가능성을 발견한다  
**우선순위:** P1  
**내용:** 첫 5개 커뮤니티 프로젝트를 적극 발굴 (Discord/X에서 Robota 사용자 찾기, 빌더 지원). Showcase 페이지가 비어있는 것은 제품이 아무도 쓰지 않는다는 신호로 읽힌다.

### PM-007: /explain 명령 — 코드 설명 원클릭

**사용자 가치:** 코드 리뷰나 레거시 이해에서 즉각적인 가치를 경험한다  
**우선순위:** P2  
**내용:** `/explain <파일경로>` — 파일을 읽고 주요 구조, 역할, 의존성을 요약. 첫 Aha Moment 경로를 단축하는 단순하지만 강력한 명령.

### PM-008: 설치 실패 진단 도구

**사용자 가치:** 설치 안 되는 이유를 혼자 알아낼 수 있다  
**우선순위:** P2  
**내용:** `robota --diagnose` 명령. Node 버전, 환경 변수, 설정 파일, 프로바이더 연결성을 자동 체크하고 문제 진단 리포트 출력. 신규 사용자 이탈의 주요 원인인 설정 오류를 자가 진단.

### PM-009: Product Hunt / HN 런칭 준비

## 결론

`@robota-sdk/agent-cli`는 기술적으로 올바른 방향으로 만들어지고 있다. 아키텍처가 견고하고, 멀티 프로바이더 지원이 실제로 작동하며, SDK 임베딩 가능성이라는 진정한 차별점이 있다.

그러나 제품으로서 채택 가능성은 아직 낮다. Node 22 진입 장벽, CJK 크래시, 빈 커뮤니티, SDK 예제 부재가 실제 사용자 획득을 막고 있다. "좋은 아키텍처"와 "사람들이 실제로 쓰는 도구"는 다른 것이다.

가장 중요한 한 가지를 꼽는다면: **SDK 임베딩 포지셔닝에 집중하고, 이를 5분 만에 체험할 수 있는 예제를 만들어라.** Claude Code와 직접 경쟁하는 CLI 게임보다, "AI 코딩 도구를 만드는 엔진"이라는 포지션이 더 크고 덜 붐비는 시장이다.
