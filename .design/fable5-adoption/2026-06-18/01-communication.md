# reference profile 채택 — 커뮤니케이션 / 포매팅 스타일

## 요약

사용자는 Claude reference profile 시스템 프롬프트를 이 저장소 하네스의 에이전트 행동 권위로 채택하기로 결정했다. **무제한 reference profile 우선권**이 적용된다. 즉 reference profile 원칙과 기존 하네스 규칙이 충돌하면 reference profile가 이긴다. 본 문서는 그중 **커뮤니케이션과 포매팅 스타일** 영역만 다룬다.

핵심 결론을 먼저 말하면, 충돌은 한 군데에서만 발생하고 그것이 결정적이다. reference profile는 "보고서, 문서, 기술 문서, 설명"을 불릿이나 번호 목록 없는 산문으로 쓰라고 요구한다. 그러나 이 저장소의 하네스는 마크다운 구조 그 자체를 기계가 파싱하는 계약으로 사용한다. SPEC.md의 필수 섹션, 백로그 frontmatter, 비교 표, common-mistakes 표, 규칙 인덱스 표가 모두 그런 구조 계약이다. reference profile를 문자 그대로 적용하면 이 계약 산출물이 깨진다.

해결 방향은 거부가 아니라 범위 분리다. reference profile 산문 우선 원칙은 **대화형 출력과 서술형 산문**(채팅 응답, PR 설명, 커밋 본문, 보고 메시지, 설명·근거 텍스트)에 최대로 적용하고, **기계가 파싱하는 구조 산출물**(SPEC.md 섹션, frontmatter, 계약 표)에는 적용하지 않는다. 사용자 지시에 따라 reference profile가 우선권을 가진다는 사실은 그대로 기록하되, 구조 산출물 예외를 구현 리스크로 명시한다. 나머지 원칙(톤, 질문 1개 제한, 모호성 선해소, 거부 시 불릿 금지, 욕설 금지, 파일 존재 확인)은 대부분 새 추가 항목이며 충돌 없이 채택 가능하다.

이 영역은 기존 `RULE-001-agent-communication-conduct.md` 초안과 겹친다. RULE-001은 "전이 가능한 약 8%만, 그것도 기존 규칙이 안 다루는 부분만 보수적으로 흡수"하는 입장이다. 본 RULE-002는 사용자 지시에 따라 정반대로 **충돌 시 reference profile 우선**을 채택하며 커뮤니케이션/포매팅만 깊이 다룬다. 두 초안은 GATE-APPROVAL에서 통합하거나 RULE-002가 RULE-001의 커뮤니케이션 항목(A)을 대체하도록 정리해야 한다.

## reference profile 원칙 추출

원본 `_SOURCE-reference.md`의 `### tone_and_formatting`(68–91행, `#### lists_and_bullets` 포함)과 `### evenhandedness`(134–147행)에서 커뮤니케이션·포매팅 관련 원칙을 모두 뽑으면 다음과 같다.

첫째, 톤은 따뜻하되 정직하다. 상대의 판단력이나 능력을 깎아내리는 가정 없이 친절하게 대하고, 반박이 필요하면 건설적으로 한다. 둘째, 설명에 예시·사고실험·비유를 쓸 수 있다. 셋째, 사용자가 먼저 욕설을 쓰거나 요청하지 않는 한 욕설을 쓰지 않으며, 그런 경우에도 절제한다. 넷째, 질문을 항상 하지는 않으며 하더라도 한 응답에 하나를 넘기지 않고, 모호한 요청이라도 먼저 처리하려 시도한 뒤에 명확화를 요청한다. 다섯째, 상대가 미성년자로 의심되지 않는 한 유능한 성인으로 가정하고 그렇게 대한다. 여섯째, 프롬프트가 파일이 있다고 암시해도 실제로 있는 것은 아니므로 직접 확인한다.

포매팅 규율은 더 구체적이다. 굵게 강조·헤더·목록·불릿 같은 과잉 포매팅을 피하고 명확성에 필요한 최소한만 쓴다. 목록·불릿·포매팅은 (a) 요청받았거나 (b) 내용이 충분히 다면적이어서 명확성에 필수일 때만 쓴다. 불릿은 사용자가 달리 요청하지 않는 한 최소 1~2문장이어야 한다. 일상 대화와 단순 질문에서는 목록 대신 산문으로 자연스럽게 답하고 짧아도 된다. 보고서·문서·기술 문서·설명에서는 사용자가 목록이나 순위를 요청하지 않는 한 불릿·번호 목록·과잉 굵게 없이 산문으로 쓰며, 산문 안의 나열은 "포함되는 것으로는 x, y, z가 있다"처럼 줄바꿈 없이 자연스럽게 녹인다. 작업을 거절할 때는 결코 불릿을 쓰지 않는다. 추가적 배려가 충격을 누그러뜨리기 때문이다.

evenhandedness 쪽에서 포매팅과 직접 닿는 부분은, 복잡하거나 논쟁적인 주제에 단순 예/아니오나 한 단어 답을 요청받으면 그 짧은 형식을 거절하고 미묘한 답을 주며 왜 간결함이 부적절한지 설명할 수 있다는 점이다.

## 채택 항목

무제한 reference profile 우선권에 따라 추출한 원칙 전부를 채택한다. 채택 형태는 새 규칙 문서 `.agents/rules/agent-conduct.md`에 "Communication & Formatting" 섹션으로 싣는 것을 권장하며, 구체적 규칙 문안은 백로그 초안의 Proposed Rule Additions에 영어로 작성했다.

대화형·서술형 출력(채팅 응답, PR 설명, 커밋 메시지 본문, 보고 메시지, 설명·근거·요약 텍스트)에는 산문 우선·최소 포매팅·불릿 1~2문장·질문 1개 제한·거절 시 불릿 금지·따뜻하되 정직한 톤을 전면 적용한다. 충돌이 발생한 "보고서/문서/기술 문서는 산문" 원칙도 이 범위에서는 그대로 따른다. 다만 기계가 파싱하는 구조 산출물에는 적용하지 않는다. 이 분리가 채택의 핵심이며, 우선권은 어디까지나 reference profile에 있되 구조 계약을 깨지 않는 선에서 최대로 적용한다는 뜻이다.

## 구현 리스크

가장 큰 리스크는 reference profile의 88행 원칙("보고서·문서·기술 문서·설명은 불릿·번호 목록·과잉 굵게 없는 산문")을 문자 그대로 저장소 전체에 적용할 때 발생한다. 이 저장소는 마크다운 구조를 사람이 읽는 문체가 아니라 기계가 파싱하는 계약으로 쓴다. 구체적으로 SPEC.md는 정해진 필수 섹션 헤더 집합을 가져야 하고(`spec-writing-standard` 스킬과 `harness:scan`이 검사), 백로그 spec-docs는 `--- status / type / tags ---` frontmatter와 정해진 섹션 구조를 요구하며, common-mistakes 표·규칙 인덱스 표·비교 표는 행/열 구조로 의미를 전달한다. 여기서 불릿·헤더·표를 산문으로 풀어버리면 `harness:scan`이 깨지고, 백로그 게이트 파이프라인이 frontmatter를 못 읽으며, 사람이 빠르게 대조하던 계약 표를 잃는다.

권장 해결책은 적용 범위를 두 갈래로 나누는 것이다. 첫째 갈래는 자유 서술형 산문, 즉 사람을 향한 대화·설명·보고다. 여기에는 reference profile 산문 우선·최소 포매팅을 전면 적용한다. 둘째 갈래는 스키마가 규정된 구조 산출물, 즉 frontmatter, SPEC.md 필수 섹션, 계약 표다. 여기서는 구조가 곧 정확성이므로 산문 강제를 적용하지 않는다. 규칙 문안에 이 경계를 명시하고, "구조가 스키마/계약으로 규정된 산출물은 예외이며 그 내부의 산문 부분에 한해 reference profile 산문 규율을 적용한다"고 적는다.

이 예외는 reference profile 우선권을 부정하는 것이 아님을 분명히 한다. 사용자 지시에 따라 충돌 시 reference profile가 우선한다는 원칙은 유지된다. 다만 reference profile 원칙 자체가 "보고서·문서"를 사람이 읽는 산문 산출물로 전제하고 있고, 기계 파싱 계약 파일은 그 전제 밖에 있다는 해석으로 둘을 양립시킨다. 만약 사용자가 이 해석조차 거부하고 구조 산출물에도 산문을 강제하길 원한다면, 그것은 하네스의 구조 불변식(harness:scan 통과, 게이트 파이프라인 동작)을 깨는 별도 대규모 작업이므로 GATE-APPROVAL에서 명시적 결정이 필요하다.

부차 리스크가 둘 더 있다. 하나는 AGENTS.md와 규칙 문서 자체가 굵게·표·번호 목록을 적극 사용한다는 점이다. 이들은 사람과 에이전트가 빠르게 라우팅하는 참조 계약 문서이므로 위 둘째 갈래(구조 산출물)로 분류해 예외 처리한다. 다른 하나는 RULE-001과의 중복·상충이다. RULE-001은 보수적 흡수를, RULE-002는 무제한 우선권을 전제하므로 GATE-APPROVAL에서 둘의 관계를 확정해야 한다. 본 초안은 RULE-002가 커뮤니케이션/포매팅 영역에서 RULE-001 항목 A를 상위 대체하도록 권장한다.
