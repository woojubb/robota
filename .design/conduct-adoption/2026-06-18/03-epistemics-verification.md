## 1. RCP 인식/검증 원칙 추출

원문에서 이 영역에 해당하는 핵심 원칙은 다음과 같다.

| ID  | RCP 원칙                                                                                                                                                                                                                                                               | 근거 (행)           |
| --- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------- |
| F1  | **변경 가능한 사실은 단언 전에 검색으로 검증한다.** 안정적·시대불변 지식(역사적 사실, 과학 원리, 완료된 사건)은 직접 답하되, 컷오프 이후 바뀔 수 있는 현재 상태(직위 보유자, 현행 정책, 현존 여부)는 검색해 확인한다. 의심스럽거나 최신성이 문제될 수 있으면 검색한다. | 158, 450            |
| F2  | **특정 제품/모델/버전/최신 기법은 답하기 전에 검색한다.** 학습 데이터의 부분적 인식은 현재 지식이 아니다. 짧은 버전형 이름("v0", "o1", "2.5")이나 신규 기법 약어도 개념이 익숙하더라도 검색 대상이다.                                                                  | 457                 |
| F3  | **인식되지 않는 개체 규칙(UNRECOGNIZED ENTITY).** 모르는 대상을 답하려면 그것이 무엇인지 알아야 하는데 특정할 수 없다면 → **검색.** 추측으로 지어내는 비용은 사용자 신뢰의 손실이다. 검색은 몇 초의 비용이다.                                                          | 458                 |
| F4  | **현재 날짜를 쿼리에 사용한다.** 실제 현재 날짜를 사용한다. "latest iPhone 2025"처럼 과거 연도를 박으면 stale 결과가 나온다 — "latest iPhone" 또는 현재 연도를 쓴다.                                                                                                   | 160, 476            |
| F5  | **검색 결과(또는 그 부재)에 대해 과신하지 않는다.** 결과의 타당성이나 결과 없음에 대해 과한 단정을 하지 않으며, **공평하게(evenhandedly)** 제시하고 성급히 결론내지 않으며 사용자가 추가 조사할 수 있게 둔다.                                                          | 164, 484            |
| F6  | **실질적 외부 주장은 출처를 인용한다.** 검색 결과에 기반한 모든 구체적 주장은 인용 태그로 출처를 단다. 답에 영향을 주는 출처만 인용하고, 충돌하는 출처는 명시한다. 원본 고품질 출처(공식 블로그, 동료심사 논문, 정부/SEC)를 집계 사이트보다 우선한다.                  | 484, 486, 1535–1543 |
| F7  | **stale 기억에 의존하지 않는다.** 컷오프(2026-01) 이후 변할 수 있는 모든 것은 검색으로 확인한다. 컷오프 언급은 관련될 때만 한다.                                                                                                                                       | 158, 460            |
| F8  | **노력을 복잡도에 맞춘다.** 단일 사실 1회, 중간 작업 3–5회, 심층 비교 5–10회. 과도/과소 검색 모두 회피. 동일·유사 쿼리 반복 금지.                                                                                                                                      | 462, 473            |

## 2. 코딩 에이전트로의 매핑 (이 레포)

소비자 챗 제품의 "웹 검색"은 코딩 에이전트에서 **"라이브러리/API/도구의 현재 동작을 문서·코드에 대조해 확인"**으로 사상된다. 이 레포에는 context7(라이브러리 문서), 다수의 vercel/cloudflare 스킬(공식 문서 우선 retrieval), `claude-api`/`run`/`verify` 등 "기억보다 문서 우선" 트리거가 이미 존재한다.

| RCP        | 코딩 에이전트 매핑                                                                                                                                                                                                                                         |
| ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| F1, F2, F7 | **단언/코딩 전에 현재 라이브러리·API·SDK·CLI 동작을 문서/소스/코드에 대조 검증한다.** 기억된(잠재적으로 stale한) 시그니처·옵션·기본값·동작에 의존하지 않는다. 메이저 버전 변경, 신규 API, 잘 모르는 패키지는 context7/공식 문서로 확인 후 코드를 작성한다. |
| F3         | 인식되지 않는 패키지명·플래그·심볼·에러 메시지는 추측하지 않고 레포 코드(grep/read)나 공식 문서로 특정한 뒤 다룬다.                                                                                                                                        |
| F4         | 외부 문서/릴리스 노트를 찾을 때 **실제 현재 날짜/현재 버전**으로 조회한다. 과거 연도·구버전을 박아 stale 답을 만들지 않는다.                                                                                                                               |
| F5         | **검증 결과를 충실히 보고한다.** "동작한다/통과한다/존재하지 않는다"를 실제 증거 없이 과신하여 단정하지 않는다. 결과의 부재(예: "그런 API는 없다")도 확인 없이 단정하지 않는다. 발견을 공평하게 제시하고 잔여 불확실성을 명시한다.                         |
| F6         | 실질적 외부 주장(API가 X를 한다, Y가 기본값이다, Z 버전부터 변경됨)에는 **출처(문서 URL, 파일 경로, 코드 라인)**를 단다. 충돌하는 출처는 명시한다. 서드파티 소스코드는 근거가 아니라 읽을 공식 문서를 가리키는 포인터일 뿐이다(`research.md`와 정합).      |
| F8         | 검증 노력을 작업 위험도에 맞춘다 — 사소한 변경엔 과한 조사를 피하고, 계약/런타임 영향이 큰 변경엔 비례적 조사를 한다.                                                                                                                                      |

핵심 한 줄: **"기억(낡았을 수 있음) 대신 문서/코드에 대조해 현재 동작을 검증한 뒤 단언·코딩하라. 검증 결과는 (성공·실패·불확실 모두) 충실히 보고하라. 실질적 외부 주장에는 출처를 달라."**

## 4. Implementation Risk (구조적 긴장)

- **R1 — RULE-001과의 소유권 중복.** RULE-001(agent-communication-conduct) 초안이 이미 "C. Epistemic discipline (thin)"과 "D. Untrusted-content hygiene (thin)"를 포함한다. 본 RULE-004는 바로 그 C 영역의 **심화 소유자**다. 두 초안이 동시에 같은 영역을 건드리면 SSOT/중복 금지 원칙 위반 위험. → **해소안: RULE-004가 인식·검증의 단독 소유자가 되고, RULE-001은 C 항목을 RULE-004로 위임(한 줄 참조)하도록 정렬.** GATE-APPROVAL에서 확정 필요.
- **R2 — process.md verification과의 경계.** `verification.md`는 **자기 변경물의 런타임 실행 증거**(build/test/behavioral/headless)를 다룬다. 본 규칙은 **외부 사실/타사 API에 대한 사전 인식 규율**이다. 두 축은 상보적이나, 배치 시 "어디까지가 verification.md, 어디부터가 신규 규칙인가"를 명확히 구분하지 않으면 중복으로 보일 수 있다. → 배치 권고는 §아래.
- **R3 — 도구 의존성.** F1~F3는 외부 검증 도구(context7, 웹 fetch, 공식 문서)의 가용성을 전제한다. 도구가 없을 때의 행동(= "불확실을 충실히 보고하고 추측하지 않음", F5로 흡수)을 규칙에 포함해야 공백이 없다.

**배치 권고:** 신규 규칙 문서 `.agents/rules/agent-conduct.md`(RULE-001이 제안한 동일 파일)에 **"Epistemic Discipline & Verification" 섹션**으로 수록하는 것이 가장 깔끔하다. 사유: (1) RULE-001과 동일 owner 문서로 모이면 conduct 계열이 한 곳에 응집, (2) `verification.md`는 self-change 런타임 증거 전용으로 유지해 경계가 선명, (3) RULE-001의 thin C/D 항목을 본 섹션으로 위임해 중복 제거. 대안으로 `process.md`(verification) 산하 신규 절도 가능하나, 이는 외부-사실 인식 규율을 self-verification 게이트에 섞어 경계를 흐린다.
