"나도 만들 수 있을까?"

에이전트 라이브러리를 가지고 있었다. 이 위에 Claude Code 같은 CLI를 올릴 수 있을까?

10개월 전, Robota SDK를 만들어 npm에 공개했다. 멀티 프로바이더 AI 엔진, 도구 시스템, 플러그인. Team 에이전트까지 개발했고, CLI도 시험해 봤다. Robota는 Cursor로 바이브 코딩하면서 개발했는데, 월 $1,000을 쓰기도 했다. 공개 후 한동안 업데이트를 하지 않았다.

당시에는 AGENTS.md 같은 개념이 없어서 코딩 에이전트에 적용해 보지 못했지만, 최근에 Claude Code를 매일 쓰면서 "나도 코딩을 도와주는 CLI를 구현할 수 있을까?" 하는 생각이 들었다.

AGENTS.md는 시스템 프롬프트다

특별한 파싱이 아니라, 대화 시작 시 시스템 메시지로 넣는 텍스트

AGENTS.md, CLAUDE.md 같은 파일은 프로젝트 루트에서 탐색한 뒤 시스템 프롬프트에 주입한다. LLM은 이 내용을 "프로젝트의 규칙"으로 인식하고 따른다.

  • 대화가 시작되면 AGENTS.md 내용을 시스템 메시지로 넣는다
  • 문서 안에 다른 .md 파일 링크가 있으면 그 파일도 참조하도록 유도한다
  • LLM은 이 규칙을 따르면서 도구를 호출하고, 결과를 받아서 다시 판단한다

코딩 에이전트의 핵심은 결국 이 세 가지다: 시스템 프롬프트 + 도구 + 실행 루프.

에이전트가 읽는 프로젝트 구조

프로젝트 폴더에 넣으면 에이전트가 자동으로 인식한다

코딩 에이전트는 프로젝트 루트의 특정 폴더를 탐색해서 스킬, 훅, 설정을 자동으로 로딩한다.

경로내용
.agents/skills/프로젝트 전용 스킬 — 슬래시 명령으로 실행 가능
.agents/rules/프로젝트 규칙 — AGENTS.md에서 참조
.agents/tasks/백로그, 진행 중 작업 추적
.claude/settings.json훅 설정, 퍼미션 allow/deny 목록
.claude/commands/Claude Code 커스텀 명령

폴더에 파일을 넣으면 에이전트가 자동으로 인식한다. 별도 설정이 필요 없다.

코딩 에이전트의 기본 도구

에이전트의 손발 — 이 6개로 코드를 읽고, 쓰고, 실행한다

도구역할예시
Bash셸 명령 실행pnpm test, git status, ls
Read파일 읽기소스 코드, 설정 파일 확인
Write파일 생성새 파일 작성
Edit파일 부분 수정함수 하나만 고치기
Glob파일 검색**/*.ts 패턴 매칭
Grep내용 검색코드에서 함수 이름 찾기

LLM이 이 도구를 자유롭게 조합해서 호출한다. 파일을 읽고, 수정하고, 테스트를 돌리고, 결과를 확인하는 — 개발자가 터미널에서 하는 일을 그대로 한다.

실행 루프의 구조

프롬프트 하나에 도구가 여러 번 돌 수 있다

한 라운드

LLM에 메시지를 보내면 응답이 온다. 응답에 도구 호출이 포함되어 있으면 도구를 실행하고, 결과를 대화에 추가한 뒤, LLM을 다시 호출한다. 도구 호출이 없을 때까지 반복한다.

멀티 라운드

사용자가 "이 파일을 읽고 버그를 고쳐줘"라고 하면:

  • 라운드 1: LLM이 Read로 파일을 읽는다
  • 라운드 2: 읽은 내용을 보고 Edit으로 수정한다
  • 라운드 3: Bash로 테스트를 돌린다
  • 라운드 4: 테스트 결과를 보고 최종 응답을 한다

스트리밍

각 라운드에서 LLM의 응답은 토큰 단위로 실시간 출력한다. 도구 실행 상태도 동시에 표시한다. 사용자는 AI가 "생각하는 과정"을 실시간으로 본다.

도구 실행의 안전장치

Bash와 Write를 아무 제한 없이 실행할 수는 없다

에이전트가 파일을 수정하고 셸 명령을 실행한다. 잘못하면 코드를 망가뜨리거나 위험한 명령을 실행할 수 있다. 그래서 도구 호출마다 퍼미션 체크가 필요하다.

3단계 평가

  • deny 목록 — 명시적으로 차단한 패턴 (예: rm -rf /)
  • allow 목록 — 미리 허용한 패턴 (예: pnpm test:*)
  • 모드 정책 — 나머지는 현재 모드에 따라 결정

4단계 모드

모드동작
plan읽기만 허용, 수정 전부 차단
default위험한 도구는 사용자에게 확인
acceptEdits파일 수정 자동 허용, 셸 명령만 확인
bypassPermissions전부 자동 허용 (위험)

훅 시스템

도구 실행, 세션 시작/종료, 사용자 입력 등 주요 시점에 끼어들 수 있다

코딩 에이전트의 동작 흐름 곳곳에 커스텀 로직을 끼워 넣을 수 있다. 도구 실행 전후뿐 아니라, 세션 시작/종료, 사용자 입력 직후에도 훅이 동작한다.

훅 이벤트시점활용 예시
PreToolUse도구 실행 전브랜치 보호, 위험 명령 차단
PostToolUse도구 실행 후자동 포매팅, 린트 실행
SessionStart세션 시작컨텍스트 로딩, 상태 체크
Stop세션 종료로그 저장, 정리 작업
UserPromptSubmit사용자 입력 후입력 전처리, 검증

훅은 셸 커맨드, HTTP 호출, 프롬프트, 에이전트 실행 등 여러 방식으로 구현할 수 있다.

스킬 시스템

폴더에 스킬 파일을 넣으면 슬래시 명령으로 실행할 수 있다

.agents/skills/ 또는 .claude/commands/ 안에 마크다운 파일을 넣으면 에이전트가 자동으로 발견하고 슬래시 명령 메뉴에 등록한다.

스킬 파일 구조

--- name: commit description: 변경 사항을 커밋 context: fork --- 스킬 프롬프트 내용...
  • name — 슬래시 명령 이름 (/commit)
  • description — 메뉴에 표시할 설명
  • context: fork — 별도 세션에서 실행 (서브에이전트)

스킬은 프로젝트마다 다르게 구성할 수 있다. 코드 리뷰, 커밋, 테스트, 문서 생성 등 반복 작업을 스킬로 만들어두면 슬래시 명령 한 번으로 실행한다.

컨텍스트 관리 문제

대화가 길어지면 AI가 앞의 내용을 잊는다

AI 모델은 한 번에 처리할 수 있는 컨텍스트 크기에 한계가 있다. Sonnet 4.5는 200K 토큰, Opus 4.6은 1M 토큰. 많아 보이지만, 코딩 에이전트는 컨텍스트에 엄청난 양의 정보를 담는다.

왜 문제인가

  • 대화 내용뿐 아니라 도구 실행 내역, 파일 내용, 명령 결과가 전부 컨텍스트에 쌓인다
  • 도구가 파일을 읽을 때마다 그 내용이 대화에 추가된다
  • 한 번의 프롬프트에 도구가 10번 돌면, 10개의 결과가 전부 컨텍스트를 차지한다
  • 모델이 허용하는 컨텍스트 크기를 넘으면 API가 에러를 낸다

코딩 에이전트가 해결해야 하는 것

  • 컨텍스트가 얼마나 찼는지 추적
  • 한계에 가까워지면 이전 대화를 요약해서 압축
  • 시스템 프롬프트(AGENTS.md)는 압축하지 않고 보존
  • 도구 출력이 너무 크면 잘라내기

이 구조를 직접 구현해 봤다

내 에이전트 엔진 위에 올렸더니 작동했다

AGENTS.md와 실행 루프만 맞춰줬더니 정말 Claude Code처럼 동작했다. 핵심은 기본 도구 5개(Bash, Read, Write, Edit, Glob, Grep)였다. 그리고 이 구조는 Anthropic API에 종속하지 않는다 — OpenAI API를 써도 동일하게 돌아갈 구조다.

기존 Robota SDK 위에 계층을 나눠서 쌓았다:

agent-cli 터미널 UI (Ink/React)
agent-sdk InteractiveSession, 커맨드, 스킬, 플러그인
agent-sessions Session, 퍼미션, 훅, 컨텍스트
agent-core Robota 엔진, 프로바이더, 도구, 실행 루프

단방향 의존성. 위에서 아래로만 의존한다. core는 아무것에도 의존하지 않는다.

플러그인 호환과 서브에이전트

생태계와 호환하지 않으면 또 하나의 고립한 도구로 끝난다

이중 경로 전략

~/.robota/ ← Robota 주 경로 ~/.claude/ ← Claude Code 호환 (우선순위 높음) .agents/skills/ ← Robota 스킬 .claude/commands/ ← Claude Code 커맨드 (호환)

서브에이전트가 필요한 이유: Claude Code의 스킬 파일에는 프론트매터에 context: fork라는 설정이 있다. 이 스킬은 메인 대화와 분리한 별도 세션에서 실행하라는 뜻이다. CLI에서 이걸 지원하지 않으면 플러그인이 의도와 다르게 동작하거나 메인 컨텍스트를 오염시킨다.

만들면서 부딪힌 문제 (1)

터미널에서 한국어를 쓰는 건 쉽지 않았다

CJK 입력

문제: 터미널 raw mode에서 한국어 IME가 정상 작동하지 않았다. 글자가 탈락하거나, 잘못된 위치에 입력되거나, Mac 터미널이 크래시하기도 했다.

원인: React의 useState가 비동기로 배칭하는데, IME는 동기로 키스트로크를 보낸다. 이 충돌이 근본 원인이었다.

해결: 기존 라이브러리(ink-text-input)를 버리고 CjkTextInput을 직접 작성. useRef로 값을 즉시 반영하고, useState는 렌더링 트리거로만 사용.

Mac 터미널 크래시 (SIGSEGV)

문제: Mac 기본 터미널(Terminal.app)에서 한국어를 입력하면 앱 자체가 크래시했다.

원인: Ink의 setCursorPosition()이 터미널 커서를 잘못된 위치로 이동시키면, IME가 그 위치에서 attributedSubstringFromRange:를 호출하다가 세그멘테이션 폴트 발생.

해결: setCursorPosition 호출을 제거. IME 후보창 위치가 왼쪽 하단에 고정되지만 크래시는 막았다.

만들면서 부딪힌 문제 (2)

데이터 구조를 바로잡으니 나머지가 풀렸다

컨텍스트 오버플로우

문제: 도구 결과가 실시간으로 누적되면서 컨텍스트가 급격히 커지고, LLM이 오류를 내며 중단.

해결: 임계값 초과 시 도구 결과를 더 추가하지 않고 "컨텍스트 문제로 누락됨" 메시지를 삽입. LLM이 이를 보고 자연스럽게 복구.

히스토리 아키텍처 리팩토링

채팅 메시지, 도구 실행, 스킬 호출 등 모든 정보가 하나의 타임라인에 남아야 했다. agent-core에 IHistoryEntry 타입을 신설하고, 히스토리에 규칙을 세웠다:

  • append-only, read-only — 수정/삭제 없음. 데이터가 없으면 fallback이 아니라 버그
  • 투명한 저장 — resume 시 가공 없이 그대로 불러옴. 변환이 필요하면 저장 로직이 잘못된 것

이 규칙 덕분에 세션 이어가기, 히스토리 표시, 이벤트 추적을 모두 단순하게 풀었다. 세션을 JSON 파일로 저장하고, resume 시 그대로 불러와서 AI 컨텍스트에 주입한다.

그래서, 지금 쓸 수 있나?

된다

실제로 코딩 작업에 사용해 봤고, 일상 개발 작업을 할 수 있는 수준이다. 이제 Robota를 Robota CLI로 개발할 수 있다.

  • Claude Code와 동일한 도구 세트
  • 스킬/플러그인 생태계 호환
  • 세션을 이어가며 며칠에 걸친 작업 가능
